# QA / E2E / Ship Boundary Clarification

**Date:** 2026-04-22
**Scope:** `plugins/me/skills/qa/`, `plugins/me/skills/e2e/`, `plugins/me/skills/ship/`

## Goal

`qa`, `e2e`, `ship`의 개념 경계를 다시 정리한다. 특히 `qa`와 `e2e`를 경쟁 관계처럼 읽히지 않게 만들고, `ship`을 검증 스킬이 아니라 출시 readiness gate로 분리해 설명한다.

## Problem

현재 문서에서는 세 가지 혼선이 있다.

1. `qa`는 이미 구현 검증 중심으로 이동했지만, 일부 문구는 여전히 버그 탐색 중심 설명을 남기고 있다.
2. `e2e`는 `qa`와 병렬인 별도 카테고리처럼 읽히지만, 실제로는 더 좁은 특수 목적 검증에 가깝다.
3. `ship`은 검증 결과를 소비하는 readiness gate인데, 다른 두 스킬과 같은 종류의 검증 스킬처럼 보일 여지가 있다.

이 상태에서는 사용자가 "언제 무엇을 써야 하는가"를 즉시 판단하기 어렵다.

## Design Principle

정리 축은 기능이나 단계가 아니라 **목적**이다.

- `qa`는 기능이 맞는지 본다.
- `e2e`는 연결이 이어지는지 본다.
- `ship`은 지금 내보내도 되는지 본다.

이 세 문장은 문서 전반의 기준 문장으로 유지한다.

## Role Model

### `qa`

`qa`는 현재 변경사항이 의도대로 동작하는지 확인하는 **기본 검증 스킬**이다.

기본 초점:
- golden path
- 핵심 edge case
- 명백한 regression

`qa`는 기본 검증 경로다. 특별한 이유가 없으면 검증 요청은 먼저 `qa`가 받는다.

### `e2e`

`e2e`는 여러 컴포넌트, 계층, 서비스 경계를 가로지르는 전체 흐름이 실제로 연결되는지 확인하는 **특수 검증 스킬**이다.

`e2e`는 일반적인 기능 검증 전체를 대체하지 않는다. 기본 검증은 `qa`가 맡고, 서비스 경계 검증이 핵심 리스크일 때만 `e2e`를 추가한다.

즉 `e2e`는 `qa`와 경쟁하는 상위 검증이 아니라, 더 좁고 더 특정한 목적의 검증이다.

### `ship`

`ship`은 기능을 다시 검증하는 스킬이 아니라, 이미 확보된 검증 근거와 운영 준비 상태를 바탕으로 현재 변경을 배포해도 되는지 판단하는 **출시 readiness gate**다.

핵심 초점:
- pre-launch evidence
- rollout readiness
- rollback readiness
- monitoring readiness

`ship`은 검증 단계 다음에 오는 판단 단계다.

## Relationship

세 스킬의 관계는 아래처럼 정리한다.

- 기본 경로는 `qa`다.
- 서비스 경계 검증이 핵심인 경우에만 `e2e`를 추가한다.
- `ship`은 `qa`와 `e2e`에서 확보한 검증 근거를 바탕으로 배포 가능 여부를 판단하는 마지막 단계다.

문서에서는 `qa vs e2e` 같은 경쟁 구도를 피하고, `qa + 필요 시 e2e` 구조로 읽히게 만들어야 한다.

## Decision Rules

사용자가 언제 무엇을 써야 하는지 바로 판단할 수 있도록, 아래 운영 규칙을 명시한다.

1. 변경사항이 의도대로 동작하는지 확인해야 하면 `qa`
2. 그중에서도 서비스 경계나 다중 계층 연결이 핵심 리스크면 `e2e`
3. 검증 결과를 바탕으로 실제 배포 가능 여부를 판단하려면 `ship`

더 짧은 표현으로는 다음이 적합하다.

- `qa`는 기본 검증 경로다.
- `e2e`는 서비스 경계 검증이 필요할 때만 호출하는 특수 경로다.
- `ship`은 검증 스킬이 아니라 출시 판단 단계다.

## `qa` vs `e2e`

`qa`와 `e2e`의 차이는 검증의 넓이가 아니라 검증의 초점으로 설명해야 한다.

`qa`는 현재 변경사항이 사용자 기대대로 동작하는지를 확인하는 기본 검증이다. 기준은 기능 중심이다. 정상 흐름이 되는지, 핵심 경계 조건이 처리되는지, 이번 변경 때문에 인접 동작이 깨지지 않았는지를 본다.

반면 `e2e`는 시스템 경계를 넘는 연결 자체가 핵심 리스크일 때만 쓴다. 여기서 중요한 질문은 "이 기능이 대체로 맞는가"가 아니라 "입력부터 최종 부작용까지 실제로 끝까지 이어지는가"다.

문서에는 아래 문장을 직접 넣는 것이 좋다.

> `qa`는 기본 검증이다.
> `e2e`는 `qa`를 대체하는 상위 검증이 아니라, 서비스 경계와 다중 계층 흐름이 핵심 리스크일 때만 추가하는 특수 검증이다.

## Recommended Wording

### `qa`

> `qa`는 현재 변경사항이 의도대로 동작하는지 확인하는 기본 검증 스킬이다. 기본적으로 golden path, 핵심 edge case, 명백한 regression을 검증한다. 배포 readiness 판단은 하지 않으며, 서비스 경계를 가로지르는 전체 흐름 검증이 핵심이면 `e2e`를 추가로 사용한다.

### `e2e`

> `e2e`는 여러 컴포넌트, 계층, 서비스 경계를 가로지르는 전체 흐름이 실제로 연결되는지 확인하는 특수 검증 스킬이다. 일반적인 기능 검증 전체를 대체하지 않으며, 기본 검증은 `qa`가 맡는다. 단일 컴포넌트 수준에서 충분히 검증 가능한 변경에는 기본적으로 필요하지 않다.

### `ship`

> `ship`은 기능을 다시 검증하는 스킬이 아니라, 이미 확보된 검증 근거와 운영 준비 상태를 바탕으로 현재 변경을 배포해도 되는지 판단하는 readiness gate다. 이 스킬은 rollout, rollback, monitoring, pre-launch 관점에서 blocker와 warning을 정리하고 최종 readiness를 판정한다.

## Editing Direction

이 스펙에 따라 문서를 수정할 때는 아래 원칙을 따른다.

1. `qa` 설명에서 버그 탐색이나 QA 엔지니어 역할을 전면 정의처럼 쓰지 않는다.
2. `e2e` 설명에서 `qa`를 병렬 경쟁 개념으로 소개하지 않는다.
3. `ship` 설명에서 기능 검증을 다시 수행하는 듯한 표현을 제거한다.
4. 세 스킬 모두 "무엇을 하지 않는가"를 짧게 적어 경계를 분명히 한다.
5. 세 스킬 관계를 설명할 때는 항상 `qa`를 기본값으로 두고, `e2e`를 예외적 특수 경로로 배치한다.

## Acceptance Criteria

다음이 만족되면 정리 목표를 달성한 것으로 본다.

- `qa`, `e2e`, `ship`이 각각 다른 질문에 답하는 스킬로 설명된다.
- `qa`가 기본 검증 경로로 정의된다.
- `e2e`가 `qa`보다 좁은 특수 목적 검증으로 정의된다.
- `ship`이 검증 스킬이 아니라 readiness gate로 정의된다.
- 문서를 처음 읽는 사람도 "언제 무엇을 쓰는가"를 세 줄 규칙으로 판단할 수 있다.
